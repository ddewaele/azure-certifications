## Module Overview

This module continues from Part 1 and covers the hands-on side of identity management in Microsoft Entra ID: creating member and guest user accounts, managing user lifecycle (reset, disable, delete, restore), assigning licenses, and organizing users into security and Microsoft 365 groups — including dynamic membership rules.

## 1. User Account Types

### 1.1 Two Types of User Accounts

| Account Type | Purpose | Created How |
|-------------|---------|-------------|
| **Member account** | For users who are **part of your organization** (employees, internal staff) | Created directly in Entra ID via "New User" |
| **Guest / External account** | For users **outside your organization** (B2B collaboration, contractors, project partners) | Created via an **invitation** sent to their email |

### Key Exam Takeaways
- Member = internal, Guest = external
- Guest accounts are created through an **invitation workflow**, not directly

## 2. Creating a Member User Account

### 2.1 Mandatory Fields (Basics Tab)

There are four mandatory fields when creating a new user:

| Field | Description |
|-------|------------|
| **User Principal Name (UPN)** | The unique identifier for the account (e.g., `alicejones@contoso.onmicrosoft.com`) |
| **Mail nickname** | Derived from the UPN by default; can be unchecked and customized (e.g., shortening "Alexandra" to "Alex", or simplifying a double-barrel name) |
| **Display name** | The name shown across the environment (Teams, email, directory) — can differ from the UPN |
| **Password** | Auto-generated by default; can be manually set. User is **forced to reset** on first login |

### 2.2 Account Enabled Toggle
- Accounts are **enabled by default** at creation
- If the user won't start immediately, **disable the account** until their start date → aligns with **Zero Trust** methodology (enable just-in-time)

### 2.3 User Properties — Why They Matter
- Every user account has a long list of optional properties: job title, department, company, manager, hire date, etc.
- **Recommendation:** complete as many properties as possible at creation time
- Properties are used later by features like **dynamic group membership rules**, conditional access, and access reviews
- **Usage location** (e.g., UAE, UK, US) is technically optional but **required before you can assign a license** — omitting it causes an error at license assignment time

### 2.4 Assignments Tab
- Roles and group memberships can be assigned at creation, but this is optional — they can be added later

### Key Exam Takeaways
- Know the four mandatory fields, especially **UPN** as the unique identifier
- Users are forced to **reset their password on first login**
- **Usage location must be set** before a license can be assigned — know the error scenario
- Completing user properties is important for dynamic groups and other downstream features

## 3. Inviting External / Guest Users

### 3.1 Invitation Workflow
1. Click "New User" → "Invite External User"
2. Provide the guest's **email address**
3. A **redirect URL** is included in the invitation email
4. Optionally add a **personal message** explaining the reason for the invite

### 3.2 Security of the Invitation Link
- The invitation link is **bound to the specific email address** it was sent to
- If someone else intercepts and tries to use the link, it will **not work** — it only works for the intended recipient

### 3.3 Guest User Properties
- Only **two mandatory fields**: email address and redirect URL
- All other properties (name, job title, company, etc.) are optional but **recommended** to complete for future feature integration

### 3.4 Assignments
- Guest users can also be assigned to **groups** and **roles** at invitation time or later

### Key Exam Takeaways
- Guest invitations are email-bound — the link is useless to anyone other than the intended recipient
- Only two mandatory fields for guest accounts vs. four for member accounts
- Complete as many properties as possible for both member and guest accounts

## 4. User Account Management (Lifecycle)

### 4.1 Management Actions Overview

| Action | How | Details |
|--------|-----|---------|
| **Reset password** | User account → "Reset Password" | For users not part of SSPR, admins can reset directly |
| **Disable account** | User overview → Edit → Uncheck "Account Enabled" | Prevents authentication; use for compromised accounts |
| **Revoke sessions** | User account → "Revoke Sessions" | Breaks all **currently active authenticated sessions** immediately |
| **Delete account** | User account → "Delete" | Moves user to "Deleted Users" — **not permanent immediately** |
| **Restore account** | Deleted Users → "Restore User" | Available for **30 days** after deletion |

### 4.2 Compromised Account Response — Correct Order
When an account is compromised, follow this sequence:
1. **Revoke active sessions** — breaks any current access immediately
2. **Disable the account** — prevents new logins
3. Investigate and remediate

### 4.3 Deleted Users — 30-Day Retention
- Deleted accounts are kept in a **soft-delete state for 30 days**
- Within 30 days → account can be **restored**
- After 30 days → account is **permanently deleted** and cannot be recovered

### Key Exam Takeaways
- Know the **30-day soft-delete window** — this is a common exam question
- For compromised accounts: **revoke sessions first**, then disable
- SSPR (from Part 1) vs. admin-initiated password reset are separate concepts

## 5. License Assignment

### 5.1 Where to Assign Licenses
- License management has moved **out of the Entra portal**
- Licenses are now managed in the **M365 Admin Center** at `admin.microsoft.com`
- This applies to both **user-level** and **group-level** license assignments

### 5.2 Assigning a License to a User
1. Go to `admin.microsoft.com` → Licenses
2. Select the license plan (e.g., M365 E5, which includes Entra Premium P2)
3. Click "Assign License" → search for and select the user
4. Before confirming, you can **toggle individual features on/off** within the plan (e.g., disable Clipchamp or Microsoft Loop while keeping everything else)
5. Optionally send an email notification to the user about the new license

### 5.3 Assigning a License to a Group
- Same process as user-level, but you select a **group** instead
- You can also toggle individual features on/off for the entire group
- ⚠️ You must have **sufficient available licenses** for all group members — otherwise you get an error

### 5.4 Usage Location Dependency
- A user **must have a usage location set** before a license can be assigned
- Different locations unlock different features — some services aren't available in all regions

### Key Exam Takeaways
- Licenses are managed in **admin.microsoft.com**, not in the Entra portal
- You can **granularly enable/disable specific features** within a license plan
- **Usage location is a prerequisite** for license assignment
- Group-based licensing requires **enough licenses for all members**

## 6. Groups in Microsoft Entra ID

### 6.1 Group Types Comparison

| Feature | Security Group | Microsoft 365 Group |
|---------|---------------|-------------------|
| **Primary use** | Manage access to applications, resources; assign licenses | Collaboration — shared mailbox, calendar, files, SharePoint |
| **Member types** | Users, devices, service principals, **other groups** (nesting) | **Users only** |
| **Group nesting** | ✅ Allowed | ❌ Not allowed |
| **Device membership** | ✅ Allowed | ❌ Not allowed |
| **Entra role assignment** | ✅ Can be enabled | ✅ Can be enabled |

### 6.2 Creating a Group — Key Settings

| Setting | Details |
|---------|---------|
| **Group type** | Security or Microsoft 365 — chosen at creation |
| **Group name** | Descriptive name (e.g., "ITSupport Level 2") |
| **Group description** | Explain who should be in this group and why — important for other admins managing the environment |
| **Entra roles can be assigned** | Yes or No — **this setting is fixed after creation and cannot be changed later** |
| **Membership type** | Assigned, Dynamic User, or Dynamic Device (device option is **security groups only**) |

### 6.3 Entra Roles Assignment to Groups — Critical Rule
- You choose **Yes** or **No** for "Microsoft Entra roles can be assigned to the group" at creation time
- ⚠️ **This setting is permanent** — once the group is created, it cannot be changed
- Plan ahead: decide whether the group needs role assignment **before** creating it

### Key Exam Takeaways
- Security groups support **users, devices, service principals, and nested groups** — M365 groups support **users only**
- The "roles can be assigned" toggle is **immutable after group creation**
- Always provide a clear **description** for groups

## 7. Group Membership Types

### 7.1 Assigned Membership
- Members are **manually added** by an administrator or group owner
- **Group owner** = a designated person responsible for managing that specific group's membership (doesn't need to be a full admin)
- Members can be added at creation or any time after

### 7.2 Dynamic Membership
- Members are **automatically added and removed** based on rules that evaluate user (or device) properties
- You **cannot manually add** members to a dynamic group — attempting to do so produces an error
- Available types:
  - **Dynamic User** — available for both Security and M365 groups
  - **Dynamic Device** — available for **Security groups only**

### 7.3 Dynamic Membership Rules — Syntax

Rules follow this pattern:
```
user.property -operator "value"
```

**Example:**
```
user.department -eq "IT"
```
→ Any user whose department property equals "IT" is automatically added to the group.

### 7.4 Combining Expressions: AND vs. OR

| Operator | Behavior | Example |
|----------|----------|---------|
| **AND** | **Both** conditions must be true | `user.department -eq "IT" and user.companyName -ne "Fabricam"` → must be in IT **and** not from Fabricam |
| **OR** | **Either** condition can be true | `(first rule) or user.city -eq "London"` → matches the first rule **or** lives in London (regardless of the first rule) |

### 7.5 How AND and OR Work Together
- **AND** = combines conditions additively — all must be true as a unit
- **OR** = creates a completely **separate, independent condition** — if either side of the OR is true, the user qualifies
- Microsoft recommends **caution with multiple OR operators** — they create entirely new rule branches and can make logic complex and hard to maintain

### 7.6 Common Operators

| Operator | Meaning |
|----------|---------|
| `-eq` | Equals |
| `-ne` | Not equals |

*(Other operators like `-contains`, `-startsWith`, `-in`, etc. are available but not covered in this transcript)*

### Key Exam Takeaways
- You **cannot manually add members** to a dynamic group — this is enforced by the system
- Know the difference between **AND** (all must match) and **OR** (independent alternative condition)
- Dynamic rules rely on **user properties** — reinforces why completing properties at account creation is important
- Dynamic Device membership is **security groups only**
