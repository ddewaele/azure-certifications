## Module Overview

This module covers two core governance tools: Azure Policy (controlling what resources can exist and where) and Role-Based Access Control (controlling who can do what). Together, Policy and RBAC form the two pillars of Azure governance — Policy governs resources, RBAC governs people.

---

## 1. Azure Policy — Why and What

### 1.1 Why Use Azure Policy?
- Control and limit what can be created in your environment
- Enforce compliance standards and regulations
- Restrict resource types, SKUs, regions, or features
- Ensure tagging standards are followed
- Enable mandatory features (e.g., backup on all VMs)

### 1.2 Common Policy Examples

| Policy | Purpose |
|--------|---------|
| **Allowed resource types** | Limit which resource types users can create |
| **Allowed VM SKUs** | Restrict which VM sizes are available |
| **Allowed locations** | Restrict resources to specific regions |
| **Add/inherit a tag** | Enforce tagging standards for billing and management |
| **Enable backup on VMs** | Ensure compliance with backup requirements |

## 2. Azure Policy — Three Core Concepts

### 2.1 The Three Steps

| Step | What It Does |
|------|-------------|
| **1. Definition** | Defines the rule — what the policy checks for. Can be built-in (hundreds available) or custom. |
| **2. Scope** | Where the policy applies — management group, subscription, or resource group. **Cannot** be applied to individual resources. |
| **3. Compliance** | Reports which resources comply and which don't. Non-compliant resources are flagged but not stopped/deleted. |

### 2.2 Policy Scope Levels

| Scope | Available? |
|-------|-----------|
| Management group | Yes |
| Subscription | Yes |
| Resource group | Yes |
| Individual resource | **No** |

### Key Exam Takeaways
- Policies can be applied at management group, subscription, or resource group — **not** individual resources
- Always choose the highest appropriate scope for least admin effort

---

## 3. Policy Behavior — New vs. Existing Resources

### 3.1 Timeline After Assignment
- Policy takes **5 to 15 minutes** to become fully effective
- After that, new resource creation is blocked if it violates the policy (deny effect is real-time at submission)
- Existing non-compliant resources are **not stopped, deleted, or modified** — they continue running

### 3.2 Compliance Reporting
- Existing resources are evaluated and flagged as **compliant** or **non-compliant** in the compliance report
- Non-compliance is informational — it tells you there's a problem but doesn't fix it automatically
- Standard compliance scan runs **every 24 hours**
- On-demand scans can be triggered via CLI: `az policy state trigger-scan`

### 3.3 The Two Behaviors Summary

| Behavior | New resources | Existing resources |
|----------|--------------|-------------------|
| **Deny effect** | Blocked immediately | Not affected — keep running |
| **Compliance report** | Evaluated within ~15 min | Evaluated within 24h cycle |
| **Remediation** | Not applicable (never created) | Can be remediated if task is defined |

### Key Exam Takeaways
- New resources = blocked in real-time by deny effect
- Existing resources = flagged in compliance report, never automatically removed
- Compliance scan is periodic (24h), not real-time

## 4. Policy Assignment Walkthrough

### 4.1 Assigning a Policy (Portal Steps)
1. Search for **Policy** in the Azure portal
2. Go to **Definitions** → find or search for a policy (e.g., "Allowed locations")
3. Click the policy → **Assign**
4. Set the **Scope** (management group, subscription, or resource group)
5. Configure **Parameters** (e.g., which locations are allowed)
6. Optionally configure **Remediation** tasks
7. Add a **Non-compliance message** (shown to users when they violate the policy)
8. **Review + create**

### 4.2 Non-Compliance Messages
- Custom message displayed to users when they try to create a resource that violates the policy
- Appears inline in the portal (if policy is at subscription/resource group level) or at submission time (if at management group level)
- Should explain what's wrong and how to comply

### 4.3 Remediation Tasks
- Can automatically fix certain types of non-compliance (e.g., add missing tags)
- Works with **deployIfNotExists** and **modify** effects
- Cannot move existing resources to a different region — manual intervention required for location-based policies
- Requires a **managed identity** to execute tasks on your behalf

## 5. Policy Initiatives

### 5.1 What Is an Initiative?
- A **group of multiple policies** bundled together into a single assignment
- Instead of applying 20, 50, or 500 individual policies one by one, bundle them into an initiative and assign once
- Follows the **least admin effort** principle

### 5.2 How Initiatives Work
1. Create or select an initiative definition (can include built-in and custom policies)
2. Assign the initiative to a scope (management group, subscription, or resource group)
3. All policies within the initiative are applied at once
4. Can reuse the same initiative across multiple scopes with tweaks

### 5.3 Built-In Initiatives
- Azure provides pre-built initiatives for common compliance frameworks: ISO 27001, NIST, CIS, PCI-DSS, etc.
- These are **free to assign** — no additional cost
- They evaluate your environment against the framework and generate a compliance dashboard

### Key Exam Takeaways
- Initiative = bundle of policies assigned as one unit
- Use initiatives for least admin effort when applying many policies
- Built-in compliance initiatives (ISO 27001, etc.) are free

## 6. RBAC — Role-Based Access Control

### 6.1 What Is RBAC?
- Controls **who** can do **what** on Azure resources
- Also referred to as **IAM** (Identity and Access Management) or **Access Control** in the portal
- Separate from Entra ID roles — RBAC is for Azure resources, Entra roles are for directory/identity management

### 6.2 Entra ID Roles vs. Azure RBAC Roles

| Aspect | Entra ID Roles | Azure RBAC Roles |
|--------|---------------|-----------------|
| **Scope** | Entra ID directory (users, groups, apps) | Azure resources (VMs, storage, networks) |
| **Examples** | Global Administrator, Application Administrator | Owner, Contributor, Reader |
| **Where managed** | Entra portal | Azure portal → IAM at any scope level |
| **Purpose** | Manage identities and directory | Manage Azure resources and access |

### 6.3 Key Principle: Separate Worlds
- Having Global Administrator in Entra does **not** automatically give you access to Azure resources
- Exception: the Global Admin can enable a setting in Entra to gain **User Access Administrator** on the root management group, then assign themselves further roles

### Key Exam Takeaway
- Entra roles ≠ Azure RBAC roles — know the distinction

## 7. The Four Built-In RBAC Roles

### 7.1 Role Comparison

| Permission | Owner | Contributor | Reader | User Access Admin |
|-----------|-------|------------|--------|------------------|
| Read resources | Yes | Yes | Yes | Yes |
| Create / modify / delete resources | Yes | Yes | No | No |
| Assign roles to others (IAM) | Yes | **No** | No | Yes |
| Create support tickets | Yes | Yes | Yes | Yes |

### 7.2 Role Details

**Owner** — the most powerful role. Can do everything including assigning roles to others. Use sparingly.

**Contributor** — identical to Owner **except** cannot assign or change roles/permissions. Cannot elevate themselves or others. The go-to role for engineers and developers.

**Reader** — view-only access. Can read all resources but cannot create, modify, or delete anything. Ideal for auditors, finance teams, managers.

**User Access Administrator** — can view resources and manage role assignments, but cannot create or modify resources. Typically used by Global Admins who need to grant access to Azure subscriptions.

### 7.3 How to Read Role Permissions (JSON)
Every role has a JSON definition with four key sections:

```json
{
  "roleName": "Contributor",
  "permissions": [
    {
      "actions": ["*"],          // what you CAN do
      "notActions": [            // what you CANNOT do
        "Microsoft.Authorization/*/Write",
        "Microsoft.Authorization/*/Delete",
        "Microsoft.Authorization/elevateAccess/Action"
      ],
      "dataActions": [],
      "notDataActions": []
    }
  ]
}
```

- `actions: ["*"]` = can do everything
- `notActions` = exceptions carved out (for Contributor: cannot write/delete/elevate authorization)
- Reader's definition: `actions: ["*/read"]` — can only read, no notActions needed

### Key Exam Takeaways
- Owner vs. Contributor: the **only** difference is permission management (IAM)
- User Access Administrator: can manage access but **cannot** manage resources
- Know how to read the JSON: actions = allowed, notActions = blocked

## 8. RBAC Scope and Inheritance

### 8.1 Where RBAC Can Be Applied

| Scope | IAM Available? |
|-------|---------------|
| Management group | Yes |
| Subscription | Yes |
| Resource group | Yes |
| Individual resource | Yes |

Note: RBAC can be applied at the **resource level** — unlike Azure Policy which stops at resource group.

### 8.2 Inheritance Rules
- Roles assigned at a **higher scope cascade down** to all children
- Assigning Contributor at the subscription level → applies to all resource groups and resources in that subscription
- Inheritance flows **top-down only** — cannot be removed at a lower level
- Roles are **additive** — if you have Reader inherited from above and get Contributor assigned directly, you get Contributor (higher wins)

### 8.3 Scope Strategy — Least Privilege + Least Admin Effort

| Scenario | Best Approach |
|----------|-------------|
| User manages all resources in a subscription | Assign role at **subscription** level |
| User manages only VMs across multiple resource groups | Assign **Virtual Machine Contributor** at subscription level (wide scope, narrow permissions) |
| User manages all resources in one project | Assign role at **resource group** level |
| User manages a single VM | Assign role at **resource** level |

### Key Exam Takeaways
- RBAC applies at all four levels (including individual resources) — unlike Policy which stops at resource group
- Widen scope + narrow role type = least admin effort with least privilege (e.g., VM Contributor at subscription level)
- Inheritance is top-down and additive

## 9. Specialized/Granular Roles

### 9.1 Why Use Them?
- The three general roles (Owner, Contributor, Reader) are broad
- Granular roles restrict access to specific resource types while allowing wider scope
- Azure has **680+ built-in roles** — filterable by category (Compute, Networking, Storage, etc.)

### 9.2 Common Examples

| Role | What It Allows |
|------|---------------|
| **Virtual Machine Contributor** | Manage VMs only — not storage, networking, or other resources |
| **Network Contributor** | Manage networking resources only |
| **Storage Blob Data Reader** | Read blob data only |
| **SQL DB Contributor** | Manage SQL databases only |

### 9.3 Custom Roles
- If no built-in role fits your needs, you can **create custom roles**
- Define specific actions and notActions in JSON format
- Assign to any scope just like built-in roles

### Key Exam Takeaway
- Use granular roles to follow least privilege: wide scope + narrow role beats narrow scope + broad role

## 10. The Global Admin Bridge (Entra → Azure)

### 10.1 The Scenario
- A Global Admin in Entra ID has no Azure resource access by default
- They need to manage a subscription (e.g., assign a role to a new project manager)

### 10.2 The Process
1. Global Admin goes to **Entra ID** → **Properties**
2. Enables the toggle: **"Access management for Azure resources"**
3. This grants them the **User Access Administrator** role at the root management group
4. They can now go to the subscription → IAM → assign roles to themselves or others
5. After done, they should disable the toggle (security best practice)

### 10.3 Why This Matters
- Contributors cannot assign roles → if the only Owner is unavailable, no one can grant access
- This is the emergency "break glass" path through the Global Admin Entra setting
- Only Global Admins have this capability

### Key Exam Takeaway
- Global Admin enabling "Access management for Azure resources" in Entra = User Access Administrator at root management group. Know this path — it's a common exam scenario.

## 11. Azure Policy vs. RBAC — The Key Distinction

| | Azure Policy | RBAC |
|---|-------------|------|
| **Controls** | What can be done (resource rules) | Who can do it (user permissions) |
| **Scope** | Mgmt group, subscription, resource group | Mgmt group, subscription, resource group, **resource** |
| **Effect** | Deny, audit, modify, deployIfNotExists | Allow / not allow actions |
| **Override** | Even an Owner is blocked by a deny policy | Policy doesn't care about who — only what |
| **Example** | "No VMs outside Europe" | "Alice can manage VMs" |

### Key Exam Takeaway
- Policy and RBAC work **together**: RBAC says Alice has Contributor access. Policy says no VMs outside Europe. Alice can manage VMs — but only in Europe. Both apply simultaneously.
