## Module Overview

This module covers the fundamentals of Microsoft Entra ID as a cloud-based identity and access management (IAM) service. It walks through the transition from on-premises Active Directory to Entra ID, hybrid identity scenarios, licensing tiers, and Self-Service Password Reset (SSPR).

---

## 1. On-Premises Active Directory — The Starting Point

### 1.1 Traditional AD Setup
- Requires a **physical server** with a **server OS** (e.g., Windows Server)
- **Active Directory Domain Services (AD DS)** is enabled as a feature on top of the OS
- Manages **users and devices** within the organization

### 1.2 Authentication & Authorization Protocols (On-Prem)
| Protocol | Purpose |
|----------|---------|
| **NTLM** | Legacy authentication |
| **Kerberos** | Ticket-based authentication |
| **LDAP** | Directory access / queries |

### 1.3 Management & Governance Structures
- **Organizational Units (OUs)** — hierarchical containers for organizing objects
- **Group Policy Objects (GPOs)** — enforce governance and configuration policies

### Key Exam Takeaways
- Know the three on-prem protocols (NTLM, Kerberos, LDAP) and that they are **not required** in the cloud
- Understand that AD DS requires manual server setup — contrast this with the automatic provisioning of Entra ID

## 2. Microsoft Entra ID — Cloud-Based IAM

### 2.1 Core Definitions

| Term | Definition |
|------|-----------|
| **Identity** | An object that can be authenticated |
| **Account** | Data associated with an identity |
| **Microsoft Entra Account** | An account created via Microsoft Entra ID or other Microsoft Cloud services |
| **Tenant** | A dedicated representation of your organization in the cloud |
| **Domain** | A unique address registered during tenant setup (e.g., `contoso.onmicrosoft.com`) |

### 2.2 How a Tenant Is Created
1. Create a Microsoft Cloud account (or already have one via M365)
2. Provide organizational details and **register a domain** (must be unique)
3. Microsoft automatically provisions a **Microsoft Entra ID service** in the back-end — no server setup, no OS config, no AD feature installation
4. The person who creates the account becomes the **first user** and receives the **Global Administrator** role (highest level of access)

### 2.3 Modern Authentication Protocols (Cloud)
| Protocol | Purpose |
|----------|---------|
| **OAuth** | Authorization |
| **OpenID Connect** | Authentication (built on OAuth) |
| **SAML** | Web-based SSO / authentication |
| **WS-Federation** | Federated authentication |

- All are **web-based** → require **HTTPS**
- NTLM, Kerberos, LDAP are **not required** in the cloud
- Legacy protocols **can** still be made available in Entra for legacy application support

### Key Exam Takeaways
- Cloud protocols vs. on-prem protocols — be able to classify each
- A tenant is **automatically** provisioned; no infrastructure management needed
- The first user always gets the **Global Administrator** role
- Legacy protocols are not needed but can be enabled for backward compatibility

## 3. Hybrid Identity — Bridging On-Prem and Cloud

### 3.1 Microsoft Entra ID Connect (Entra Connect)
- Tool used to **sync on-prem users and devices** into the cloud
- Eliminates the need to **recreate** all on-prem identities in the cloud
- After sync, on-prem users can access **cloud-based resources** (e.g., office.com)

### 3.2 Hybrid Authentication Flow (Example)
1. **User 1** (on-prem) logs into their device → authenticated by **on-prem AD**
2. User 1 opens `office.com` → traffic is redirected to **Entra ID**
3. Entra ID authenticates User 1 (recognizes them because they were synced via Entra Connect)
4. Entra sends a signal back → user is granted access to the cloud portal

### 3.3 Sync Direction
- Entra Connect syncs **one direction** by default: on-prem → cloud
- A **cloud-only user** (User 2, created directly in Entra) does **not** automatically appear in on-prem AD
- To sync certain attributes **back** to on-prem, you must enable **writeback options**

### 3.4 Writeback Options
| Writeback Type | What It Does |
|---------------|-------------|
| **Password writeback** | Syncs cloud password changes back to on-prem AD → user avoids managing two passwords |
| **Device writeback** | Syncs device details and settings back to on-prem |

### 3.5 Without Password Writeback — The Problem
- User resets password in the cloud → change stays in the cloud only
- On-prem password remains unchanged → user now has **two different passwords**
- Enabling password writeback **solves** this by syncing the new password down to AD

### Key Exam Takeaways
- Know the role of Entra Connect and its **default one-way sync direction**
- Understand the writeback concept — especially **password writeback** and why it matters
- Hybrid user = exists both on-prem and in the cloud
- Cloud-only user = exists only in Entra ID, not in on-prem AD

## 4. Microsoft Entra ID Licensing Plans

### 4.1 Plan Comparison

| Feature | Free | Premium P1 | Premium P2 | Entra Suite |
|---------|------|-----------|-----------|-------------|
| Single Sign-On (SSO) | ✅ | ✅ | ✅ | ✅ |
| Up to 500,000 objects | ✅ | ✅ | ✅ | ✅ |
| Multi-Factor Authentication (MFA) | ✅ | ✅ | ✅ | ✅ |
| Advanced features | ❌ | ✅ | ✅ | ✅ |
| Privileged Identity Management (PIM) | ❌ | ❌ | ✅ | ✅ |
| Identity Governance | ❌ | ❌ | ✅ | ✅ |

### 4.2 Key Points
- If you have **any** Microsoft Cloud account, you likely already have the **Free plan**
- **P1** adds conditional access, self-service group management, and more
- **P2** adds **Privileged Identity Management (PIM)** and **Identity Governance**
- **Entra Suite** is the most comprehensive tier
- Full feature comparison available at Microsoft's Entra plans and pricing page

### Key Exam Takeaways
- Know which features are free vs. P1 vs. P2
- PIM and Identity Governance → **P2 or higher**
- Free plan already includes SSO, MFA, and up to 500K objects

## 5. Self-Service Password Reset (SSPR)

### 5.1 What Is SSPR?
A feature that allows users to **reset their own passwords** without administrator intervention, reducing helpdesk load.

### 5.2 SSPR Enablement Modes
| Mode | Description |
|------|------------|
| **None** | Disabled for everyone |
| **All** | Enabled for all users in the organization |
| **Selected** | Enabled for a **single selected group** only |

- ⚠️ You can only select **one group** at a time — selecting a new group **replaces** the previous one
- **Group nesting is allowed** — use a parent group containing nested sub-groups to work around the single-group limitation

### 5.3 SSPR Authentication Methods
Users must register **one or two** authentication methods during onboarding. Available methods include:

- Authenticator app (one-time passcode or push notification)
- Code generated from the authenticator app
- Security questions
- Other methods listed in the Entra authentication methods settings

### 5.4 Security Questions — Details
- Can be used **for SSPR only** (not for general MFA)
- Users register between **3 and 5** questions
- Challenge can require answering **3, 4, or 5** of the registered questions
- Microsoft provides **20 predefined questions** — but many are vulnerable to **social engineering** (e.g., "What's your favorite food?", "What company did you work for?")
- **Custom questions** can be created to improve security

### 5.5 Admin-Specific Rules
- Admins are **always required to register two authentication methods**, even if regular users are only required to register one
- This is a built-in security measure — not configurable

### 5.6 SSPR by License Tier
| Capability | Free | Premium P1 | Premium P2 |
|-----------|------|-----------|-----------|
| Users who can reset | Cloud-only users | Cloud + hybrid users | Cloud + hybrid users |
| Can reset if they **know** current password | ✅ | ✅ | ✅ |
| Can reset if they **forgot** password | ❌ | ✅ | ✅ |
| Can reset if account is **locked out** | ❌ | ✅ | ✅ |

### Key Exam Takeaways
- SSPR allows **None / All / Selected** — and Selected supports **only one group** (but nesting is allowed)
- Admins always need **two** authentication methods for SSPR
- Security questions are **SSPR-only**, not general MFA
- Free tier SSPR only works if the user **already knows** their current password → limited usefulness
- P1/P2 add forgot-password and locked-out-account scenarios

## 6. Quick Reference — On-Prem AD vs. Microsoft Entra ID

| Aspect | On-Prem Active Directory | Microsoft Entra ID |
|--------|--------------------------|-------------------|
| Hosting | Physical server on-prem | Cloud-based (automatic) |
| Setup | Install server + OS + AD DS | Register account → automatic provisioning |
| Protocols | NTLM, Kerberos, LDAP | OAuth, OpenID Connect, SAML, WS-Federation |
| Structure | Hierarchical (OUs, GPOs) | Flat (users, groups, roles) |
| Governance | Group Policy Objects | Conditional Access, PIM, Identity Governance |
| Transport | Internal network | HTTPS (web-based) |
