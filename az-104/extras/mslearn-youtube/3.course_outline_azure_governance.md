## Module Overview

This module covers the foundational architectural components of Azure: how resources are organized from the global infrastructure level (regions, availability zones) down through management groups, subscriptions, resource groups, and individual resources. It also covers governance tools including resource locks, tags, and cost management.

## 1. Azure Regions & Availability Zones

### 1.1 Regions
- Azure has **60+ regions** across **140+ countries**, with more coming
- A region is a geographic area containing one or more data centers connected by a low-latency network
- Not all features/services are available in all regions — check regional availability before deploying
- Explore regions at `aka.ms/global-infrastructure`

### 1.2 Choosing a Region
| Factor | Why It Matters |
|--------|---------------|
| **Proximity to users** | Reduces latency — deploy close to your end users |
| **Data residency / compliance** | Laws may require data to stay in a specific country (e.g., UK data laws) |
| **Availability options** | Some regions have 3 availability zones, others only 1 |
| **Feature availability** | Not all Azure services are available in every region |
| **Cost** | Pricing varies by region — some are more expensive to operate |

### 1.3 Region Pairs
- Most regions have a **paired region** at least **300 miles apart**
- Protects against natural disasters and regional outages
- Azure prioritizes recovery of one region in each pair during widespread outages
- Full list available in Azure regions documentation

### 1.4 Availability Zones
- Physically separate data centers **within a single region**
- Some regions offer **3 availability zones**, others only 1
- Deploying across zones provides redundancy — if one zone fails, the others remain operational

### Key Exam Takeaways
- Region pairs are **300 miles apart** minimum
- Not all regions have 3 availability zones — check before deploying
- Region selection affects cost, compliance, latency, and feature availability

## 2. Azure Resource Hierarchy

### 2.1 The Four Layers

```
Management groups     ← Governance container (policies, RBAC)
  └── Subscriptions   ← Billing boundary + security boundary
        └── Resource groups   ← Logical container for related resources
              └── Resources   ← VMs, VNets, LBs, FW, Apps, Storage...
```

Every resource must live inside a resource group. Every resource group must belong to a subscription. Subscriptions can optionally be organized inside management groups.

### Key Exam Takeaway
- Know this hierarchy cold — it's the backbone of Azure governance and appears in many exam questions

## 3. Subscriptions

### 3.1 What Is a Subscription?
- The mechanism for **creating and paying for** Azure resources
- Without a subscription, you cannot create resources (even with an Azure account)
- Acts as both a **billing boundary** (one bill per subscription) and a **security boundary** (access is scoped to the subscription)

### 3.2 Billing Boundary
- You receive **one bill per subscription** at the end of each month
- Bill reflects only the resources consumed within that subscription
- If you need separate bills for different departments → use separate subscriptions
- You can drill down into bills using cost analysis to see per-resource, per-region, per-service breakdowns

### 3.3 Security Boundary
- Users with access to one subscription **cannot** access resources in another subscription unless explicitly granted
- Subscriptions isolate access — useful for separating departments, projects, or environments

### 3.4 Subscription Types

| Type | Details |
|------|---------|
| **Free trial** | $200 credit for 30 days. If credit runs out or time expires, resources become inaccessible. Resources remain available (read-only) for up to 12 months. Can upgrade to pay-as-you-go. |
| **Pay-as-you-go** | Billed monthly for consumed resources. No charge for having the subscription itself — only for usage. Common for individuals and small deployments. |
| **Cloud Solution Provider (CSP)** | For SMEs working with Microsoft partners. Partner manages billing and subscription setup. |
| **Enterprise Agreement (EA)** | For large organizations. Committed spend over 1–3 years. Microsoft may suggest number of subscriptions needed. |
| **Student** | $100 credit for 12 months. Requires verification of student/educational status. |

### 3.5 Multiple Subscriptions per Account
- A single account can have many subscriptions (e.g., Dev, Test, Production)
- Subscriptions can be assigned to individual users or entire departments
- Multiple users can be members of the same subscription

### Key Exam Takeaways
- Free trial: **$200 for 30 days** — runs out on credit exhaustion OR time expiry
- Student: **$100 for 12 months**
- Subscriptions are free — you only pay for resources inside them
- One subscription = one bill (want separate bills? → separate subscriptions)
- Subscription = billing boundary + security boundary

## 4. Management Groups

### 4.1 Purpose
- Organize and govern **one or more subscriptions**
- Apply policies, RBAC, and compliance standards at a level above subscriptions
- **Inheritance flows downward** — anything applied at a management group level cascades to all child subscriptions and their resources

### 4.2 Structure
- Every tenant starts with a **root management group** (always exists)
- You can create sub-management groups up to **6 levels deep** (root level does not count)
- Maximum **10,000 objects** (management groups + subscriptions) under a single root

### 4.3 Hierarchy Example
```
Root Management Group
  ├── HR
  │     └── Apps
  ├── IT
  │     └── Production
  │           ├── Geo Region 1
  │           └── Geo Region 2
  └── Marketing
```

### 4.4 Why Use Management Groups?
- Apply a new compliance standard to HR **only** → assign policy at the HR management group level → all HR subscriptions inherit it automatically
- Avoids applying the same policy to multiple subscriptions individually
- **Least administrative effort** — a recurring exam theme

### 4.5 Limits

| Limit | Value |
|-------|-------|
| Max depth of sub-management groups | **6 levels** (excluding root) |
| Max objects under root | **10,000** |

### Key Exam Takeaways
- **6 levels deep** (root doesn't count) and **10,000 objects** — memorize these
- Inheritance flows **top-down only**
- Policies and RBAC assigned at a management group cascade to all children
- Use management groups for **least administrative effort** when governing multiple subscriptions

## 5. Resource Groups

### 5.1 What Is a Resource Group?
- A **logical container** for Azure resources (like a folder for files)
- Every resource must belong to exactly one resource group
- Ideal approach: group all resources for a **single project** into one resource group

### 5.2 Rules and Constraints

| Rule | Detail |
|------|--------|
| **Cannot be renamed** | Must delete and recreate with the correct name |
| **No nesting** | Cannot create a resource group inside another resource group |
| **Multi-resource-type** | Can hold VMs, storage accounts, web apps, etc. all together |
| **Region is for metadata only** | The resource group has a region, but resources inside can be in any region |
| **Deletion cascades** | Deleting a resource group deletes ALL resources inside it (via Azure Resource Manager) |

### 5.3 Organization Strategy

**Recommended: by project**
```
RG-WebProject
  ├── VM (web server)
  ├── Storage account
  └── Web app
```
One resource group per project → when the project ends, delete the resource group → all resources are cleaned up automatically. Least admin effort.

**Not recommended: by resource type**
```
RG-VMs       → all VMs
RG-Storage   → all storage accounts
RG-WebApps   → all web apps
```
Harder to manage lifecycle — you'd need to manually find and delete individual resources across multiple groups when a project ends.

### 5.4 Resource Group Region vs. Resource Region
- The resource group is placed in a region (e.g., UK South) — this stores **metadata** about the group
- Resources inside can be in **any region** (e.g., a VM in East US inside a resource group in UK South)
- This is allowed but generally not recommended unless there's a specific need

### 5.5 Moving Resources Between Resource Groups
- Resources can be moved from one resource group to another
- Not always straightforward — cross-region moves may have constraints
- Some resources may need to be deleted and recreated rather than moved

### Key Exam Takeaways
- Resource groups **cannot be renamed** and **cannot be nested**
- Deleting a resource group **deletes everything inside it**
- Resource group region ≠ resource region (metadata vs. actual deployment)
- Group resources **by project** for easiest lifecycle management

## 6. Resource Locks

### 6.1 What Are Resource Locks?
- Prevent accidental deletion or modification of critical resources
- Must be **removed before** a locked resource can be deleted or modified

### 6.2 Lock Types

| Lock Type | Can Read? | Can Modify? | Can Delete? |
|-----------|-----------|-------------|-------------|
| **Read-only** | Yes | No | No |
| **Delete** | Yes | Yes | No |

- **Read-only** = most restrictive — no changes, no deletion
- **Delete** = allows modifications but prevents deletion

### 6.3 Where Locks Can Be Applied
- Subscription level
- Resource group level
- Individual resource level
- ⚠️ **Cannot** be applied at the management group level

### 6.4 Multiple Locks
- Multiple locks can be applied to the same resource
- Different users/admins can each add their own lock
- All locks must be removed before the resource can be deleted

### Key Exam Takeaways
- Locks **cannot** be applied at the management group level — only subscription, resource group, and resource
- Read-only lock blocks both modify AND delete
- Delete lock blocks only delete — modification is still allowed
- Lock must be removed before deletion can proceed

## 7. Tags

### 7.1 What Are Tags?
- **Key-value pairs** applied to resources for labeling and organization
- Used for filtering resources in billing, reports, and resource management
- Every resource can have **multiple tags**

### 7.2 Common Tag Examples

| Key | Value |
|-----|-------|
| `Department` | Dev, HR, IT, Marketing |
| `Owner` | IT Admin, Alice Jones |
| `Project` | WebProject, Migration2026 |
| `CostCenter` | CC-1234 |
| `Environment` | Production, Staging, Dev |

### 7.3 Tags in Billing
- Subscription bills can be filtered by tags
- Complements other filters: service name, region, resource group, resource type
- Tags appear during resource creation (Tags tab) and can be added/modified after creation

### Key Exam Takeaways
- Tags are key-value pairs — not hierarchical
- Tags are used for **cost management filtering**, not for access control
- Resources can have multiple tags
- Tags are available for every resource type

## 8. Cost Management

### 8.1 Factors Affecting Cost

| Factor | Detail |
|--------|--------|
| **Resource type** | Different resources cost different amounts (VM vs. App Service vs. Storage) |
| **VM size/compute** | Within the same resource type, different sizes have different prices |
| **Region** | Same resource can cost more in one region than another (e.g., Brazil South > US regions) |
| **Data transfer** | **Ingress (inbound) is free** in most scenarios. **Egress (outbound) has a cost.** |

### 8.2 Cost Optimization Options

| Method | Saving | Detail |
|--------|--------|--------|
| **Reserved instances** | Up to **72%** | Commit to 1 or 3 year term for a resource |
| **Azure Hybrid Benefit** | Varies | Reuse existing on-prem licenses in the cloud |
| **Budgets & alerts** | Prevention | Set spending limits per project/subscription — alerts notify you when approaching the threshold (does NOT stop spending) |
| **Azure Advisor** | Recommendations | Provides cost-saving recommendations based on your usage patterns |

### 8.3 Cost Analysis
- Available under Subscription → Cost Analysis
- Shows total cost breakdown by: service name, region, resource group, resource type, tags
- Forecast cost shows projected spending based on current usage patterns

### 8.4 Budgets and Alerts
- Budgets set a spending target for a subscription or project
- Alerts **notify** when you approach the limit — they do **not** automatically stop resource consumption
- Useful for project cost governance

### Key Exam Takeaways
- **Ingress is free, egress costs money**
- Reserved instances save up to **72%**
- Budget alerts **notify only** — they don't stop spending
- Azure Advisor provides cost optimization recommendations
- Region affects pricing — identical resources cost different amounts in different regions

## 9. Creating a Resource — The Four Basics

Every time you create a resource in Azure, you're typically asked for four things:

| Field | Description |
|-------|------------|
| **Subscription** | Which subscription pays for this resource |
| **Resource group** | Which container to place the resource in |
| **Name** | A unique name (unique within subscription, or sometimes globally unique) |
| **Region** | Where to physically deploy the resource |

### Key Exam Takeaway
- Know these four fields — they appear in virtually every resource creation wizard

## 10. Limits and Quotas

### 10.1 Where to Find Them
- **Documentation:** Search "Azure subscription limits and quotas" on learn.microsoft.com
- **Portal:** Subscription → Settings → Usage + quotas

### 10.2 Key Points
- Every subscription has limits on how many of each resource type can be created
- Some limits are **adjustable** — request an increase via support ticket
- Some limits are **hard-set** — cannot be changed
- If you hit a limit, you can create the resource under a **different subscription** as a workaround

### Key Exam Takeaway
- Know where to find limits (portal: Subscription → Usage + quotas)
- Some limits are adjustable, some are not
- Management group limits: 6 levels deep, 10,000 objects
